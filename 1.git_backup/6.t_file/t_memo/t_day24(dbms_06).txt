1. 서브쿼리(Subquery)
	하나의 쿼리 내에 작성하는 또 다른 쿼리
	서브쿼리의 위치에 따라 종류를 구분한다
	서브쿼리는 메인 쿼리의 컬럼을 사용할 수 있지만 메인 쿼리는 서브쿼리의 컬럼 사용 불가
	소괄호 안에 작성한다

	- FROM 절 : IN LINE VIEW(인라인 뷰)
	- SELECT 절 : SCALAR(스칼라)
	- WHERE 절 : SUB QUERY(서브쿼리)

1) 인라인 뷰(INLINE VIEW)
	FROM 절 안에 또 다른 쿼리를 작성한다
	서브쿼리의 결과를 가상 테이블처럼 사용한다

2) 스칼라 서브쿼리
	SELECT 절 안에 또 다른 쿼리를 작성한다
	단일 컬럼, 단일 행을 반환한다(1개의 값으로 사용)

3) 서브쿼리
	WHERE절 또는 HAVING 절 안에 또 다른 쿼리를 작성한다


4) 연관 서브쿼리
	서브쿼리가 메인쿼리 컬럼을 가지고 있는 경우(메인쿼리 참조)

5) 비연관 서브쿼리
	서브쿼리가 메인쿼리 컬럼을 가지고 있지 않은 경우(독립 실행)

6) 단일행 서브쿼리 : 결과가 1개일 때
	사용 연산자 : =, >, <, >=, <=, <>

7) 다중행 서브쿼리 : 여러 행을 반환할 때
	사용 연산자 : IN, ALL, ANY, EXISTS

+) EXISTS
	SELECT *
	FROM 테이블 A
	WHERE EXISTS (SELECT 1 FROM 테이블 B WHERE 조건 );

+) ALL : 전부
   ANY : 하나라도(SOME)
   EXISTS : 있냐 없냐
   IN : 포함이냐 아니냐

2. VIEW	
	기존의 테이블은 그대로 놔둔 채 필요한 컬럼들 및 새로운 컬럼을 만든 가상 테이블
	실제 데이터가 저장되는 것은 아니지만 VIEW를 통해서 데이터를 관리할 수 있다

1) VIEW를 사용하는 이유
	- 보안성 : 기존의 쿼리문이 보이지 않고, 권한을 가진 사람들만 접근할 수 있도록 따로 제한할 수 있다
	- 독립성 : 다른 곳에서 원본 테이블에 접근하지 못하도록 하는 성질
	- 편리성 : 긴 쿼리문를 짧게 만드는 성질

2) VIEW 문법
	- VIEW 생성
	CREATE VIEW 뷰이름 AS 쿼리문;
	
	- VIEW 사용
	SELECT * FROM 뷰이름;

	- VIEW 삭제
	DROP VIEW 뷰이름;

3. DCL : 데이터 제어어
	데이터베이스에 접근하는 권한을 제어하는 명령어
	
	GRANT	계정에 권한을 부여하는 명령어
		GRANT 권한 [ON 테이블명] TO 계정명;

	REVOKE 	계정에 권한을 회수하는 명령어
		REVOKE 권한 [ON 테이블명] FROM 계정명;

- 계정 만들기
	SYS AS SYSDBA 접속 계정 생성 가능
	CREATE USER 계정명 identified by 비밀번호;

- 권한 부여하기
	grant create session to 계정명;
	grant create table, create view, resource to 계정명;

	create session		데이터베이스에 로그인 할 수 있는 권한
	create table		테이블 생성 가능한 권한
	create view		뷰 생성 가능한 권한
	resource		여러 객체(시퀀스, 프로시저 등) 생성 가능한 권한

- 현재 로그인한 계정의 권한 확인하기
	select * from user_tab_privs_recd;

- 현재 로그인 한 계정이 부여한 권한 확인하기
	select * from user_tab_privs_made;

- 권한 회수하기
	revoke 권한 [on 테이블명] from 계정명;

- 계정 삭제하기
	drop user 계정명;

4. TCL : 트랜잭션 제어어
1) 트랜잭션(Transaction)
	데이터베이스에서 하나의 논리적 작업 단위
	하나의 작업 묶음 세트
	여러개의 sql을 한번에 성공하거나 전부 취소하는 단위

	
update account set money = money - 10000 where name = '영선';
update account set money = money + 10000 where name = '해준';
	
2) commit과 rollback관계
	commit		확정(저장)
	rollback	취소(되돌림)

3) 트랜잭션의 4대 특징 ★★★ ACID
	Atomicity 원자성	: 전부 성공 또는 전부 실패
	Consistency 일관성 : 데이터 규칙 유지
	Isolation 고립성 : 동시에 해도 간섭 x
	Durability 영속성 : COMMIT 하면 날라가지 않음
	
5. 소프트웨어 아키텍처 디자인 패턴
	소프트웨어 시스템을 효율적이고 견고하게 설계하기 위한 구조적인 틀
	소프트웨어를 구성하는 고수준 구조(High-level Structure)
	모듈 간 역할과 관계, 데이터 흐름을 정의
	확장성, 재사용성, 유지보수성 향상

1) 계층형 구조(Layered Pattern)
	위에서 아래로 층층이 쌓은 구조
	[화면(ui)]
	↓
	[비즈니스로직]
	↓
	[데이터 베이스]

	각 층이 역할이 정해져있음
	위에서 아래로만 요청

- 장점 : 역할 분리 쉬움, 테스트하기 좋음, 구조 이해 쉬움
- 단점 : 단계가 많아지면 느려질 수 있음

2) 클라이언트-서버 패턴(Client-Server Pattern)
	요청하는 쪽 + 처리하는 쪽 구조
	[클라이언트] -> 요청 -> [서버]
	[클라이언트] <- 응답 <- [서버]

- 장점 : 구조 단순, 인터넷 서비스에 필수
- 단점 : 서버 터지면 다 멈춤, 서버에 부하 집중

3) 이벤트 기반 구조(Event-Driven Pattern)
	이벤트 발생 -> 관련 모듈 실행
	비동기 방식
	자동 반응 구조

- 장점 : 유연함, 연결 느슨함
- 단점 : 흐름 파악 어려움, 디버깅 힘듦

4) 마이크로 서비스(Microservices Pattern)
	큰 시스템을 작은 서비스로 쪼갠 구조
	회원서비스
	주문서비스
	결제서비스
	배송서비스
	
	서비스마다 독립적
	DB도 따로 가짐

- 장점 : 독립 배포 가능, 일부만 고쳐도 됨, 장애 격리
- 단점 : 구조 복잡, 관리 어려움

5) MVC Pattern
	화면 + 처리 + 데이터 분리
	Model <-> Controller <-> View

- 장점 : 역할 분리 깔끔, 유지보수 좋음
- 단점 : 작은 서비스에서는 과함

6) MVVM Pattern
	MVC + 자동 연결 강화 버전
	View <-> ViewModel <-> Model

- 장점 : UI 코드 깔끔, 자동 동기화
- 단점 : 처음 배우기 어려움

7) Pipe and Filter Pattern
	공장 라인처럼 단계별 처리
	입력 -> 필터1 -> 필터2 -> 필터3 -> 출력

	각 단계가 독립

- 장점 : 재사용 쉬움, 구조 깔끔
- 단점 : 실시간 처리 느림

6. MVC
	M(Model)
	V(View)
	C(Controller)

1) 패턴
	패턴 : 반복되는 형태/모양
	- 자주 나오는 문제를 사람들이 검증한 방식으로 반복해서 해결하다보니
	  이럴 때 이렇게 코드 구현하는 것이 좋더라 하는 것들을 약속처럼 굳어진 것이 설계 패턴
	=> 남들이 정해놓은 잘 되는 구조를 따라 쓰는 약속

2) MVC
	Java에 MVC 문법/기능이 있는 것은 아니다
	MVC는 프레임워크 기능이 아닌 설계 방식

	M = Model : 데이터/비즈니스로직(핵심처리)
		테이블에서 조회된 결과값을 담기 위한 변수들이 선언된 클래스
		클래스명 뒤에 DTO, VO라는 문자열을 붙여준다
		VO(Value Object) : 테이블을 보고 그대로 만든 객체
		DTO(Data Transfer Object) : 화면에 결과를 한번에 묶어서 전달할 객체		

	V = View : 화면(UI), 사용자에게 보여지는 부분
		사용자에게 보여질 화면을 구성하는 부분
		선언된 CRUD 메소드를 사용하는 부분

	C = Controller : 요청을 받아서 Model과 View 연결/제어
		JSP에서 더 자세히 알아볼예정
		DAO(Data Access Object) : 테이블에 접근할 수 있는 메소드들이 선언된 클래스

3) Java + DB 설계 관점
(1) DB 중심 설계(예전방식, JPA 이전)
	DB에서 테이블을 먼저 만들어놓고 그 테이블에 맞춰서 JAVA 클래스를 만드는 방식		
	DB가 기준이고 JAVA가 따라감

	- DB구조가 바뀌면 JAVA 코드가 크게 흔들림
	
(2) Java 중심 설계(요즘방식, JPA)
	Java에서 객체(클래스)를 먼저 설계하고
	그 설계를 기반으로 테이블이 만들어지거나 매핑되는 방식
	+) JPA (Java Persistence API) : JAVA 객체와 DB 테이블을 자동으로, 체계적으로 연결해주는 표준 기술
		설계가 더 깔끔해지고 유지보수성도 좋아지고 MVC같은 구조도 자연스럽게 연결됨

7. JDBC(Java Database Connectivity)
	자바에서 DB(오라클, MySQL 등)에 접속해서 
	조회(SELECT), 삽입(INSERT), 수정(UPDATE), 삭제(DELETE)를 할 수 있게 해주는 자바 표준 API

1) JDBC 전체 흐름
	Driver 로딩	
		DB 회사마다 통신방식이 달라서 그걸 번역해주는 드라이버를 로딩한다
		Class.forName("드라이버클래스");
	Connection 얻기
		DB와 연결된 통로
		연결 성공 : dbms와 정상 연결
		연결 실패 : 주소/계정/비밀번호/포트 등 문제 가능
	PreparedStatement 생성
		쿼리(SQL)를 관리해주는 객체
		SQL문을 준비해두고 값을 안전하게 넣어서 실행할 수 있다
	.executeQuery() / .executeUpdate()
		executeQuery() : SELECT 전용 => 결과가 테이블이다 ResultSet으로 받음
		executeUpdate() : INSERT, UPDATE, DELETE 전용	=> int 영향받은 행 수
					1이면 1행 성공, 0이면 실패
	ResultSet 받기(SELECT 일 때만)
		SELECT 결과(테이블)를 한 줄 씩 읽는 커서(cursor)
		next()를 해야 다음 행(레코드)로 이동하면서 읽을 수 있음
	.close()
		DB 연결 객체들은 무조건 닫아야함
		역순으로 닫는다
		rs.close()
		ps.close()
		conn.close()
