+) var와 let 차이
	var는 ES6 이전에 주로 사용하던 변수 선언 키워드, 함수 스코프를 가진다
	var로 선언된 변수가 함수 내에서만 지역변수로 작동하고 함수 외부에서는 전역변수로 작동된다

	let, const는 ES6에서 도입된 변수 선언 키워드. 블록 스코프를 가진다
	let, const keyword로 선언된 변수가 {}중괄호로 둘러싸인 블록 내에서만 유효한다는 것을 의미한다

+) var keyword 단점
	var keyword를 사용햐여 만든 변수는 함수 level 스코프를 지원한다
	그러므로 변수를 선언할 때는 let keyword를 사용하는 것을 권장한다

var
	함수 level 소코프 : 변수가 만들어진 함수 영역의 전체에서 유효함(함수 내 어디서든 접근 가능)
	재선언 가능 : 같은 스코프 내에서 동일한 변수 이름으로 다시 선언 가능
	초기화 된 접근 가능 : 변수가 선언되기 전에 접근하면 undefined 값을 가짐

let
	block level 스코프 : 변수가 만들어진 {} 영역에서만 유효함(중괄호 내에서만 접근 가능)
	재선언 불가능 : 같은 스코프 내에서 동일한 변수 이름을 다시 선언하라 수 없음
	초기화 전 접근 불가능 : 변수가 선언되기 전에 접근하면 참조에러(Reference Error)가 발생함

+) 일시적 사각지대(TDZ : Temporal Dead Zone)
   	let, const로 선언된 변수는 선언과 동시에 접근이 불가능한 구간(TDZ)에 놓임
   	따라서 let, const로 선언된 변수는 초기화 전 접근이 불가능함!!(RefferenceError)

  	var로 선언된 변수는 호이스팅 + undefined 초기화 되므로 에러가 발생하지 않음
  	따라서 var로 선언된 변수는 초기화 전 접근이 가능함(단, 타입이 undefined)

+) 호이스팅
	자바스크립트 함수가 실행되기 전에 함수 내부에 필요한 변수 값들을 모아 유효 범위의 최상단에 선언하는 것을 말한다
	다시 말해, 함수가 실행되기 전에 자바 스트립트 Parser가 함수 실행전에 해당 함수를

	함수와 연산자에 전달되는 값은 자동 형번환
	수학관련 연산자가 전달받은 값은 숫자 type으로 자동 형변환
	alert(), prompt() 전달받은 값은 해당 값의 자료형과 상관없이 문자열로 자동 형변환

- 문자형 변환
	alert(value)	// value는 문자형이어야한다
	// value가 문자형이 아닌 숫자타입이나 논리타입 등 다른 자료형의 값을 전달받으면 자동으로 문자형으로 변환된다
	String(value)	// String() 함수를 호출해 전달받은 값을 문자열로 변환 가능

	Java에서 문자열 형번환 => 다른 type의 값 + ""

- 숫자형 변환
	alert(6/3)	=> "6"/"3" 문자열이 숫자형으로 자동변환된 후 연산이 수행된다 => 2
	Number(value) 	=> Number()함수를 호출해 전달받은 값을 숫자형으로 명시하여 변환 가능

= 불린형 변환	
	Boolean(value)
		// 숫자 0. 반문자열, null, undefined, NaN => 비어있는 값(Falsy) 결과는 false
		// 비어있는 값을 제외한 모든 값 (Truty) 결과는 true

7. 대화상자
	web page에서 사용자와 상호작용하기 위해 사용되는 간단한 대화상자를 만드는 함수

	- alert() : 단순한 정보를 사용자에게 표시하기 위해 사용된다
		사용자가 획인 버튼을 클릭하기 전까지는 아무런 작업도 할 수 없다

	- confirm() : 사용자에게 "예" or "아니오" 선택지를 제공하는 대화상자
		사용자가 선택을 하고 나면 해당 선택에 따라 다른 작업을 수행할 수 있다

	- prompt() : 사용자에게 입력받을 수 있는 같을 제공하는 대화상자
		사용자가 입력할 수 있는 칸이 있고 "확인" or "취소" 선택지가 존재하여 선택에 따른 다른 작업을 수행할 수 있다

8. 연산자(Operator)
1) 최우선연산자
	Java와 동일

2) 단항연산자
	Java와 동일
	연산자를 기준으로 피 연산자(항1개)가 1개인 것을 의미
	
	+. -
	++, --
	!

3) 이항연산자
	연산자를 기준으로 피 연산자가 2개(항2개)인 것을 의미

- 산술 연산자(+ = * / %)
	Java와 동일
	+ 문자와 문자 연결, 문자와 숫자 연결, 숫자와 숫자 연산
		문자와 숫자와 숫자	문자 연결
		숫자와 숫자와 문자	숫자끼리 연산 문자열 연결

- 할당 연산자
	Java와 동일
	+=, -=, *=, /=, %=

- 쉬프트 연산자
	Java와 동일
	<<, >>, ~
	피연산자 >> 비트수 : 피연산자를 명시한 비트수만큼 이동시킨다

- 관계 연산자
	Java와 동일
	>, <, >=, <=

	==	동등 비교 연산자, 값만 같은지
	===	일치 비교 연산자, 값과 타입이 같은지
		'1' == 1	true
		'1' === 1 	false
	!=	부등 비교 연산자, 값만 다른지
	!==	불일치 비교 연산자, 값과 타입이 다른지

- 논리 연산자
	|| 논리합(or)	둘 중 하나라도 true면 true
		조건식 || 값 사용되었을경우 조건식의 값이 false일 때 뒤의 값이 출력된다

	&& 논리곱(and)	둘 다 true면 true
		선행 조건식이 true이면 뒤에 있는 값이 실행된다
		선행 조건식이 false이면 뒤에 있는 문장을 실행하지 않는다(성능을 위함)

	! 부정(not)	!true일 때 false, !false일 때 true

	?? 병합연사자	null 또는 undefined인 경우에만 오른쪽 값을 반환한다

	?. 옵셔널 체이닝(Optional Chanining)
			객체에서 undeifned나 null에 접근할 경우 에러가 발생한다
			안전하게 객체의 프로퍼티에 접근하기 위해서 사용한다

- 대입연산자
	Java와 동일

11. 프로그래밍 패러다임(Programming Paradigm)
- 패러다임 : 문제를 어떻게 바라보고, 어떤 방식으로 해결할 지에 대한 사고 틀

1) 명령형 programming(Imperative Programming)
	어떻게(How) 할 지를 computer에게 하나하나 지시
	progrem 실행 흐름과 상태(State) 변화가 중요

	1단계 -> 2단계 -> 3단계 -> 결과

(1) 절차지향 programming(Procedural Programming)
	programming 함수(절차) 단위로 나눔
	위에서 아래로 순차적으로 실행
	전역 변수 사용이 많음

- 장점
	구조가 단순하고 이해하기 쉬움
	하드웨어 가까워 처리속도 빠름
	시스템, 임베디드 개발에 적합

- 단점
	규모가 커질수록 코드 얽힘(스파게티 코드 : 실행 흐름이 얽혀서 수정하면 다른 부분이 망가지는 코드)
	유지보수의 확장 어려움
	data 보호가 어려움

(2) 객체 지향 programming(OOP)
	캡슐화	: data + method를 묶어 접근 제한
	상속	: 기존 class를 재사용하여 확장
	다형성	: 같은 method 다른 동작
	추상화	: 공통적인 특징만 뽑아 설계

	객체(Object)간의 상호작용 중심
	현실 세계 모델일에 강함
	코드 재사용성 ↑

- 장점
	유지보수 용이
	확장성 좋음
	대규모 project에 적합함

- 단점
	설계가 복잡
	실행 속도가 절차지향보다 느릴 수 있음
	객체를 남용 시 오히려 복합해짐

2) 선언형 programming(Declarative Programming)
	무엇(What)을 할지만 설명
	내부 동작 방식은 system이 맡김
	상태 변화 최소화

(1) 함수형 programming(Function Programming)
	순수함수(Pure Function)
		같은 입력 -> 항상 같은 출력
		부작용(side effect) 없음

	불변성(Immutability)
		data 변경이 없음
		새로운 data 생성

	함수는 값이다
		함수 전달, 반환 가능

- 장점
	버그 발생 가능성 감소
	병렬 처리에 유리
	테스트, 디버깅 쉬움

- 단점
	러닝 커브 높음
	상태 관리가 필요한 경우 불편
	절차적 사고에 익숙하면 어렵게 느껴짐

+) 패러다임 비교
	절차지향 : 속도 중심
	객체지향 : 구조/유지보수 중심
	함수형 : 안정성/병렬처리 중심

+) 패러다임 = 코드 사고 방식, 방법론 = 개발과정 관리 방식

3) software 개발 방법론
	개발을 구조화된 방식으로 효율적으로 수행하기 위한 체계적인 접근법

- 폭포수 모델(Waterfall)
	요구사항 -> 설계 -> 구현 -> 테스트 -> 유지보수
	단계별 순차 진행

	장점 	구조 명확, 문서화 잘 됨, 관리 쉬움
	단점 	변경에 매우 취약, 초기에 요구사항이 정확해야함
	예시 	공공/국방/금융시스템

- 애자일(Agile)
	짧은 반복 주기(스프린트)
	고객 피드백 적극 반영
	
	장점	변화에 강함, 빠른 결과물
	단점 	문서 부족 가눙, 관리 능력 중요
	예시	완벽한 계획보다 빠른 피드백이 중요한 경우, 요구사항이 자주 바뀌는 servie(모바일앱, 웹서비스, 스타트업 서비스)

- 나선형 모델(Spiral)
	반복 개발 + 리스크 분석
	점진적 개선

	장점	위험요소 사전제거, 안정성 높음
	단점 	비용 증가, 관리 복잡
	예시	항공관제 시스템, 대형금융시스템, 신기술 기반 프로젝트 => 리스크가 큰 시스템(리스크 분석이 핵심)

- RAD	
	빠른 프로토타이핑
	사용자 피드백 즉시 반영
	
	장점 	개발 속도 빠름, 고객 만족도 ↑	
	단점 	품질 저하 위험, 대규모 project 부적함
	예시	빠른 결과물이 필요한 시스템(사내 업무 관리시스템, 단기 이벤트용 웹서비스, 프로토타입 시스템 등)

- V-모델
	개발 단계 <-> 테스트 단계	1:1 대응
	
	장점	결함 초기 발견, 품질 보장
	단점 	유연성 부족, 테스트 비용 큼
	예시	품질/안전이 매우 중요한 시스템(의료시스템, 자동차제어시스템, 항공시스템 등) => 테스트 중심의 개발(오류발생이 큰사고인경우)

- XP(Extreme Programming)
	TDD
	지속적 통합
	페어 프로그래밍

	장점	코드 품질 매우 높음, 빠른 변경 반영
	단점 	팀 역량 의존, 높은 집중도 필요
	예시	코드 품질이 매우 중요한 시스템(금융거래 시스템, 걸제 시스템, 백엔드 서버 등) => 버그 최소화, 팀 실력이 중요함






	방법론		특징				장점			단점
	폭포수모델		단계별진행, 문서기반			이해하기 쉬움, 문서화잘됨	변경에 비유연
		
	애자일		반복적 개발, 고객 피드백 중시		유연성, 빠른 피드백		계획 미흡시 혼란 가능
	
	나선형모델		반복 + 리스크 관리			위험감소, 점진적 개선		복잡한 관리, 비용 증가 가능
	
	RAD		빠른 프로토타이핑, 요구사항반영빠름		속도빠름, 고객의견반영용이	품질저하, 대형 프로젝트에 부적합
	
	V-모델		개발과 테스트 병렬수행   		품질보장,결함 조기 발견		테스트 비용 높고 유연성 부족
	
	XP(익스트림)	테스트주도개발(TDD), 지속적통합		코드품질향상, 신속반영		팀 역량/협업에 의존, 집중 필요


4) 개발 방법론 선택 기준
	project 크기 : 소규모 RAD, XP
	유연성 필요 여부 : 요구사항이 자주 바뀐다면 에자일
	풀질요구 : 높은 품질을 요구한다면 V-모델, XP
	시간 : 빠른 결과물 필요하다면 RAD
	리스크 : 리스크가 크다면 나선형 모델
