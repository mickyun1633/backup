1. Web server와 WAS
	클라이언트와 server의 동신은 HTTP 약속으로 요청과 응답을 주고받을 수 있다

1) HTTP Web Server(웹 서버)
	웹 어플리케이션을 실행하기 위한 server이다
	클라이언트가 보낸 요청이 정적이 컨텐츠를 요구하는지 동적인 컨텐츠를 요구하는지 구분한다
	정적인 컨텐츠를 요청한다면 미리 준비한 웹 문서를 response에 담아 보내주지만
	동적인 컨텐츠를 처리할 수 없다
	즉, 정적 페이지를 처리하는 서버이다
	Java에서 사용하는 웹 서버는 대표적으로 Apache가 있다

	- 정적 컨텐츠 : 이미 만들어져 있는 파일, 요청이 들어와도 내용이 변하지 않음, 서버는 파일을 그대로 전달만 하면됨 
	ex) HTML file, CSS file, image file(jpg, png), 이용약관 페이지, 사용설명서 페이지 등

	- 동작 방식
	(1) 클라이언트가 / index.html 요청
	(2) 웹 서버가 해당 파일을 찾음
	(3) 파일 내용을 그대로 응답에 담아 전송

2) WAS(Web Application Server)
	웹 서버와 동일하게 웹 어플리케이션을 실행하기 위한 서버이며, 웹 컨테이너(서블릿 컨테이너)라고 불린다
	웹 서버와 다르게 정적, 동적 컨텐츠를 모두 처리할 수 있다
	단, 정적 컨텐츠 처리는 웹 서버에 비해 성능이 떨어진다
	동적 컨텐츠를 처리하기 위해 여러 연산 및 DB 접근을 수행하며 Java를 이용한다
	Java에서 사용하는 WAS는 대표적으로 Tomcat이 있다

	- 동적 컨텐츠 : 요창할 때마다 결과가 달라짐, 연산이 필요함, DB 접근이 필요함
	ex) 게시판 목록, 로그인 후 마이 페이지, 쇼핑몰 주문 내역

	- Web server보다 무거운 이유
		동적 처리는 자바코드 실행, DB 조회, 비즈니스 로직 수행, 결과 HTML 생성 과정이 필요하기 때문이다

3) Apache Tomcat
	WAS로 분류되고 Apache와 Tomcat을 합쳐놓은 형태이다
	즉, 일반적인 WAS와 달리 Web Server를 포함하고 있다
	Apache Tomcat의 Web Server는 정적인 요청을 처리하며 동적인 요청이 들어왔을 때는 웹 컨테이너에게 요청을 전달한다
	그러면 웹 컨테이너는 동적 컨텐츠를 처리하여 웹 서버에게 응답을 보낸다
	Web server는 클라이언트에게 응답을 보낼 때 컨테이너에게 받은 결과를 포함시켜 보낸다

4. Servlet(서블릿)
	servlet은 Java에서 제공하는 클래스, 기술이며 동적인 웹 페이지를 만들기 위해 사용한다
	Java 코드 안에 HTML 을 작성하는 형태이며 사용자의 요쳥방식에 따라 doGet(), doPost() 등의 메소드가 실행된다
	HttpServlet 클래스를 상속받아 사용한다

1) Servlet 처리 흐름
	Java 표준 사양(현재는 Jakarta Servlet) 기반으로 동작한다

- 클라이언트가 요청 전송(GET / POST)
- Web server -> Servlet 컨테이너 전달
- 컨테이너가 해당 servlet 객체 생성(최초 1회)
- service() -> doGet() 또는 doPost() 호출
- Java 코드 실행
- HTML 생성 후 응답 반환
	
	servlet 객체는 요청마다 새로 생성되는 것이 아니라 하나를 만들어서 여러 요청을 처리한다!

	요청해도 내용 변화가 없음, 빠르고 단순

- 동적 페이지(Dynamic)
	동일한 page라고 해도 그 때 그 때 내용이 달라질 수 있는 web page를 의미
	요청마다 결과 달라짐, Java 연산 필요, DB 접근 가능

Q. servlet객체가 하나만 생성되면 멀리쓰레드 환경에서 어떤 문제가 생길 수 있을까?
	서블릿은 기본적으로 요청마다 새롭게 생성되지 않는다
	서블릿 컨테이너는 서블릿 객체를 1개 생성, 여러 요청을 여러 쓰레드로 동시에 처리
	하나의 객체를 여러 쓰레드가 공유한다

= Race Condition	
	여러 사용자가 동시에 접근하면 같은 인스턴스 변수를 동시에 수정할 수 있다
	=> 데이터 손실

- 데이터 오염
	인스턴스 변수를 요청 데이터 저장용으로 쓰면 다른 사용자의 데이터가 섞일 수 있다

- Thread Safety 문제
	서블릿은 기본적으로 Thread-safe 하지 않다

	사용자가 우리 서버 주소 위에 /MyServlet을 작성하면 현재 서블릿이 요청을 받게된다

(2) 기본생성자

	super() => HTTpServlet의 기본 생성자를 의미

(3) doGet()와 doPost()
	HTTpServlet 클래스에 존재한다 => 오버라이딩
	이전까지는 우리가 만든 메소드를 직접 호출해서 실행했지만 이제부터는 아니다
	각 메소드는 사용자의 요청에 따라서 알아서 실행된다
	get방식으로 요청이 오면 doGet()가 실행되고, post 방식으로 요청이 오면 doPost()가 실행될 것이다
	각 메소드는 요청을 받기 위해서 request, 응답을 보내기 위해서 response 매개변수가 만들어져 있다

	★ 중요
	- doGet() -> GET 요청 처리
	- doPost() -> POST 요청 처리
	이 메소드는 우리가 직접 호출하지 않는다
	서블릿 컨테이너 (ex : Apache Tomcat)가 요청 방식에 따라 자동 호출된다


- doGet()

	protected void doGet(HttpServletRequest request, HttpServletResponse response) throws ServletException, IOException {
		// TODO Auto-generated method stub
		response.getWriter().append("Served at: ").append(request.getContextPath());
	}


	HttpServletRequest request, HttpServletResponse response 매개변수 : 요청과 응답을 담을 매개변수sssssl

	throws ServletException, IOException : 예외던지기

	respnse : 응답을 클라이언트에게 전달할 객체

	response.getWriter() : 응답에 텍스트 데이터를 작성할 수 있게 해줌

	.append("Served at: ") : 단순 문자열 출력

	request : 요청된 객체 

	request.getContextPath() : 현재 웹 어플리케이션의 경로(/day01)


- doPost()

	protected void doPost(HttpServletRequest request, HttpServletResponse response) throws ServletException, IOException {
		// TODO Auto-generated method stub
		doGet(request, response);
	}


	GET과 POST를 구분하지 않고 동일 처리할 경우 doPost() 안에서 doGet() 호출
	조회인 경우 GET, 데이터 저장/ 수정 등 POST를 사용하는 것이 일반적이다

심화복습
1. 기본 용어
1) JSP(Java Server Pages)
	동적인 웹 페이지를 개발하기 위한 웹 프로그래밍 기능
	JAVA언어를 사용하여 server측에서 웹 Pages들을 생성해 웹 브라우저로 전송해준다

- 장점
	짧은 코드로 동적인 웹 페이지를 생성할 수 있다
	기본적인 예외는 자동으로 처리된다
	많은 확장 라이브러리를 사용할 수 있다
	쓰레드 기반으로 실행되어 시스템 자원을 절약해준다


- 활용사례
	JSP는 기업용 자바 기술의 집합체인 Java EE(Java Platform, Enterprise Edition)의 핵심요소이며
	Java EE는 대한민국 정부 표준 프레임워크의 근간이다

	정부처, 공기업주도의 사업 등 대규모 기업용 시스템 구축에 사용된다
	클라우드 시대가 되면서 구글 앱 엔진이나 아마존 웹 서비스(AWS)에서도 지원하면서 활용 폭이 더 넓어졌다

2) 정적 웹페이지와 동적 웹페이지
	정적웹페이지 : 웹 서버에 저장되어있는 파일을 그대로 웹 브라우저에 전송해 출력하는 가장 기본적인 웹 페이지
	동적페이지 : 동일한 페이지라도 내용이 달라질 수 있는 웹 페이지
		서버가 클라이언트의 요청을 해석하여 가장 적절한 웹페이지를 그때 그때 생성해서 보내주는 기술

3) 탄생배경
	웹을 동적으로 만들기 위한 고대의 자바기술인 자바 애플릿(Java Applet)이 있다
	자바 애플릿은 웹에서 실행되도록 설계된 자바 어플리캐이션을 통쨰로 웹 브라우저로 전송한 후 자바 가상 머신을 탑재한
	웹 브라우저가 이를 실행하는 방식으로 구동한다

	어플리케이션 전체가 클라이언트에 다운로드 된 후 실행된는 자바 애플릿은 속도나 보안성, 유연성등에서 한계가 있었고
	서버측에서 실행되는 Servlet이 등장했다
	
	서블릿은 클라이언트의 요청을 받으면 서버에서 처리된 후 응답으로는 결과값만 보내주는 구조이다
	웹 브라우저(클랑이언트) --- 요청 ---> 웹 어플리케이션 서버(웹서버) --- 요청 ---> 서블릿 컨테이너(서블릿)
	웹 브라우저	<--- 응답 ----		웹서버	       <--- 응답 ---

	서블릿은 .java 자바파일을 컴파일한 클래스 파일(.class) 형태로 이를 실행하고 관리해주는 런타임을 서블릿 컨테이너라고 한다
	대표적인 서블릿 컨테이너로는 아파치 톰캣이 있다

7. JSP(Java Server Pages)
	Java를 기반으로 만든 웹 프로그래밍 기술이다
	서블릿의 불편한 점을 개선하고자 HTML에 Java 코드를 작성할 수 있게 만들었으며, 동적인 페이지를 만드는데 특화되어있다
	JSP도 서블릿 클래스처럼 웹 컨테이너가 관리하며, 웹 컨테이너가 JSP를 실행시킬 때 서블릿으로 변환하여 실행된다
	즉, JSP는 내부적으로 서블릿으로 변한되어 실행된다

1) get, post 방식
- get
	데이터 전달방식 : 주소(URL)에 데이터를 쿼리 스트링 형태로 추가하여 서버로 전송
	구조 : https://www.naver.com/page?key=value&key=value

	속도 : URL에 데이터를 추가해 전송하므로 요청속도가 빠름
	길이 제한 : URL 길이에 따라 데이터 전송 크기가 제한됨(일반적 2048자)
	데이터 노출 : 데이터가 URL에 그대로 노출되므로 민감한 정보(비밀번호, 개인정보 등)를 전송하면 안됨
	캐싱 및 북마크 : 브라우저가 URL을 캐싱하거나 북마크에 저장할 수 있음, 동일한 요청을 여러번 반복하는 경우 적합하게 사용됨
	보안 : HTTPS를 사용하지 않으면 데이터가 네트워크를 통해 전송될 때 평문으로 노출
		브라우저 히스토리와 서버로그에 URL이 저장되므로 민감한 데이터는 부적합함

- post
	데이터 전달방식 : 데이터가 요청(request) 본문(body)에 포함되어 서버로 전송
	서버와 클라이언트간의 데이터가 URL에 드러나지 않음

	속도 : GET 방식보다 느리지만 대량의 데이터를 처리할 수 있음
	길이 제한 : 이론적으로는 제한이 없으나 웹 서버나 웹 브라우저 설정에 따라 제한될 수 있음(몇 MB)
	데이터 노출 : 데이터가 요청 본문에 포함되므로 URL에는 나타나지 않음
	보안 : HTTPS 사용 시 데이터가 암호화되어 전송되므로 중간자 공격(MTTM)으로부터 봏보
		HTTPS를 사용하지 않으면 POST 데이터도 평문으로 노출
		서버에서 추가적인 보안조치를 하지 않으면 SQL Injection, xss 등의 공격에 취약함
		CSRF 공격에 취약 -> CSRF 토큰 등의 추가 보호가 필요함
	캐싱 : 브라우저가 POST 요청을 캐싱하지 않음 -> 데이터를 다시 전송하지 않도록 설계

2) Web.xml의 요청 흐름
	(1) 클라이언트가 request 요청을 보낸다
	(2) 웹서버가 사용자가 request 요청을 보낸 URL을 검사하여 어떤 서블릿이 request를 처리해야하는지를 결정한다
	(3) 요청이 정적 페이지를 필요로 한다면 웹 서버 자체적으로 처리하여 응답을 클라이언트에게 보내준다
	(4) 요청이 동적 페이지를 필요로 한다면 웹 서버 서블릿 컨테이너의 도움을 받는다
	(5) 서블릿 컨테이너는 web.xml 파일에서 URL에 해당하는 서블릿을 찾아서 실행한다
	(6) 서블릿은 request 요청을 처리하며 서블릿 컨테이너에게 전달하고 서블릿 컨테이너는 웹서버로 전달한다
	(7) 최종적으로 웹 서버는 클라이언트에게 응답을 보낸다
